## 构建系统概览

项目采用 pnpm workspaces 多包架构，聚合 apps/server（Hono API）、apps/web（React + Vite）与 packages/shared（共享类型与校验逻辑），通过根级 npm scripts 统一编排构建、开发与测试流程。

### 核心工具链
- 包管理: pnpm workspaces（pnpm-workspace.yaml + 根 package.json 的 workspaces 字段）
- TypeScript 编译: 各子包独立 tsconfig.json，server 使用 tsc -p tsconfig.json，web 使用 tsc -b && vite build 联合编译
- 开发服务器: server 用 tsx watch，web 用 vite dev，根脚本通过 concurrently 并行启动
- 容器化: 多阶段 Dockerfile（build → api + web），Docker Compose 编排双服务
- 运行时: Node.js ≥20（ES2022 / NodeNext ESM）

### 依赖关系与构建顺序
packages/shared (纯 TS 库, 输出 dist/index.js)
    ↑
apps/server (依赖 @mcp-debug/shared via file: protocol)
apps/web   (依赖 @mcp-debug/shared via file: protocol)

根脚本强制先构建 shared：build:server 显式执行 npm run build:shared && npm run build --workspace=apps/server，确保类型与运行时产物可用。

### 容器镜像策略
Dockerfile 定义三个 stage：
- build: node:22-alpine + python3/make/g++（编译 better-sqlite3 原生模块），执行完整 workspace 构建并 npm prune --omit=dev
- api: 仅包含 server 产物 + shared/dist，暴露 8787 端口，运行 node apps/server/dist/index.js
- web: nginx:1.27-alpine 反向代理静态资源，从 build stage 拷贝 apps/web/dist

docker-compose.yaml 将两个 target 拆分为 api 和 web 两个 service，通过 volume 持久化 SQLite 数据文件，并通过 HEALTHCHECK 实现健康检查与依赖启动顺序。

### 本地部署脚本
deployment/deploy.sh 封装 docker compose 常用操作（up/down/restart/logs/status），自动检测 docker compose v2 并生成 .env 配置文件。

### TypeScript 配置约定
- 所有包统一 target: ES2022，server 使用 module: NodeNext，web 使用 module: ESNext + Bundler 解析
- shared 包启用 declaration: true 输出类型声明，server 关闭 declaration 以减小体积
- web 使用 noEmit: true，由 Vite 负责打包；server 直接输出到 dist/

### 开发者工作流
- 安装依赖: pnpm install
- 全量构建: pnpm build:server 或 pnpm build:web
- 同时启动前后端开发: pnpm dev
- 单独启动后端: pnpm dev:server
- 单独启动前端: pnpm dev:web
- 容器化部署: ./deployment/deploy.sh up
- 运行单测: pnpm test:server

### 约束与注意事项
- 修改 packages/shared 后必须重新构建再启动 server/web，否则运行时可能缺少最新类型
- Docker 构建需要联网下载依赖，可配合缓存优化；better-sqlite3 在 musl 环境下需编译原生模块
- 环境变量通过 .env.example 模板注入，生产部署需复制并调整 deployment/.env
